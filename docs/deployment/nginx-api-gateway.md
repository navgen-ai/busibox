# Nginx API Gateway Configuration

This document describes the nginx reverse proxy pattern used across all Busibox environments (local, test, production) to provide unified API access through a single domain.

## Overview

All Busibox services are accessible through a single domain with SSL:

| Path | Service | Description |
|------|---------|-------------|
| `/portal/` | AI Portal | Main frontend application |
| `/agents/` | Agent Manager | Agent management UI |
| `/api/authz/` | AuthZ API | Authentication & authorization |
| `/api/ingest/` | Ingest API | Document processing |
| `/api/search/` | Search API | Semantic search |
| `/api/agent/` | Agent API | AI agents |
| `/api/llm/` | LiteLLM | LLM gateway |

## Benefits

1. **Single Origin**: All services share the same origin, eliminating CORS issues
2. **Unified SSL**: One certificate covers all services
3. **Simple Client Config**: Frontends only need one base URL
4. **Firewall-Friendly**: Only ports 80/443 need to be exposed

## Environment-Specific Configuration

### Local Development

- **Domain**: `https://localhost`
- **SSL**: Self-signed certificates (generated by `scripts/setup/generate-local-ssl.sh`)
- **Config Files**: 
  - `config/nginx-local.conf`
  - `config/nginx-sites/default.conf`

To generate SSL certificates:

```bash
bash scripts/setup/generate-local-ssl.sh
```

To trust the certificate on macOS:

```bash
sudo security add-trusted-cert -d -r trustRoot -k /Library/Keychains/System.keychain ssl/localhost.crt
```

### Test Environment

- **Domain**: `https://test.busibox.example.com` (or internal IP)
- **SSL**: LetsEncrypt (if public) or self-signed (if internal)
- **Config Location**: `/etc/nginx/` on the nginx container/LXC

### Production Environment

- **Domain**: `https://busibox.example.com`
- **SSL**: LetsEncrypt with auto-renewal
- **Config Location**: `/etc/nginx/` on the nginx container/LXC

## Ansible Integration

The nginx role in `provision/ansible/roles/nginx/` should be updated to:

1. Deploy the same routing pattern as local development
2. Use LetsEncrypt for SSL in production
3. Configure all API paths under `/api/<service>/`

Example Ansible variables:

```yaml
nginx_domain: busibox.example.com
nginx_ssl_provider: letsencrypt  # or self-signed for internal

nginx_upstreams:
  - name: authz_api
    server: "{{ hostvars['authz']['ansible_host'] }}:8010"
  - name: ingest_api
    server: "{{ hostvars['ingest']['ansible_host'] }}:8002"
  - name: search_api
    server: "{{ hostvars['search']['ansible_host'] }}:8003"
  - name: agent_api
    server: "{{ hostvars['agent']['ansible_host'] }}:8000"
  - name: litellm
    server: "{{ hostvars['llm']['ansible_host'] }}:4000"
  - name: ai_portal
    server: "{{ hostvars['portal']['ansible_host'] }}:3000"
  - name: agent_manager
    server: "{{ hostvars['portal']['ansible_host'] }}:3001"
```

## Client Configuration

### Frontend Applications (Next.js)

Environment variables for Next.js apps:

```env
# All APIs accessed via nginx proxy
NEXT_PUBLIC_API_BASE_URL=https://busibox.example.com

# Specific service URLs (relative paths)
NEXT_PUBLIC_AUTHZ_URL=/api/authz
NEXT_PUBLIC_INGEST_URL=/api/ingest
NEXT_PUBLIC_SEARCH_URL=/api/search
NEXT_PUBLIC_AGENT_URL=/api/agent
NEXT_PUBLIC_LLM_URL=/api/llm
```

### Backend Services

Internal services should still use direct container/host names for internal communication:

```env
# Internal service-to-service (no nginx)
AUTHZ_BASE_URL=http://authz-api:8010
INGEST_API_URL=http://ingest-api:8002

# External/browser access (via nginx)
NEXT_PUBLIC_API_URL=https://busibox.example.com
```

## Health Checks

Nginx health endpoint: `GET /health` (returns "healthy")

Individual service health checks are available at:
- `/api/authz/health/live`
- `/api/ingest/health`
- `/api/search/health`
- `/api/agent/health`
- `/api/llm/health/liveliness`

## Troubleshooting

### Service Not Available

If nginx returns 502 Bad Gateway:
1. Check if the service is running: `docker ps`
2. Check service logs: `docker logs local-<service>`
3. Verify upstream DNS resolution: `docker exec local-nginx nslookup <service-name>`

### SSL Certificate Issues

For local development with self-signed certs, browsers will show a warning. To bypass:
- Chrome: Type `thisisunsafe` on the warning page
- Firefox: Click "Advanced" → "Accept the Risk and Continue"
- Or add the certificate to your system trust store

### CORS Errors

If you see CORS errors:
1. Ensure you're using the nginx proxy URL, not direct service URLs
2. Check that the frontend is configured with the correct base URL
3. Verify SSL is working (mixed content can cause CORS-like errors)
