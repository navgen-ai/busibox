# What it actually took to install busibox on Ubuntu 24.04

Timeline of a fresh install on 2026-04-23, starting from: an Ubuntu 24.04 host with rsync'd repo at `~/maigent-code/busibox`, no prior state, all docker/rust/python deps already installed from a previous run.

## Headline

**`busibox-quick` cannot install busibox unassisted on a clean machine.** The sequential installer exits with "Installation complete!" but leaves every service either broken or unstarted. After applying eight separate workarounds, all 19 containers reach `healthy` and `https://localhost/portal/api/health` returns 200.

## Themes

The 15 issues in this directory cluster into four themes:

### 1. The secret-handling pipeline is broken end-to-end

- Issue #008: `busibox-quick` creates the vault but leaves every `CHANGE_ME_*` placeholder in place. Services literally start with `POSTGRES_PASSWORD=CHANGE_ME_POSTGRES_PASSWORD`.
- Issue #009: The ansible task that generates `.env.dev` from the vault-backed template doesn't actually run (include_role tag inheritance bug + incomplete template).
- Issue #011: Where ansible *does* inject secrets for the compose subprocess, it covers `POSTGRES_PASSWORD` but not `POSTGRES_USER` — so shell env leaks in.
- Issue #002: When the Makefile's vault helper can't decrypt, it substitutes empty strings into compose commands and lets compose error with misleading messages.

Fix direction: secrets should be auto-generated by `busibox-quick` for local-dev, env generation should fail loudly on missing values, and the compose subprocess should run under a tight explicit env allowlist (`env -i` + pass-through).

### 2. The user's shell environment contaminates everything

- Issue #001: Leftover `POSTGRES_USER`/`POSTGRES_PASSWORD` in `~/.bashrc` (common from unrelated Postgres work) is the #1 reason this install breaks. Every `docker compose` invocation picks up these shell exports, substituting them over `.env` and ansible-injected values.

Fix direction: preflight check in the installer that refuses to proceed when conflicting exports exist; sanitize the compose subprocess env.

### 3. Compose / install requires variables that aren't in `.env.dev` and nobody writes them

- Issue #012: `${VLLM_MODEL:?…}` on a profile-gated vllm service breaks every install that doesn't deploy vllm — and `scripts/llm/generate-model-config.sh`, which the error message recommends, only writes a YAML file, not something compose can read.
- Issue #009 (same as above): `.env.dev` is empty after install.
- Issue #010: `core-apps` needs a sibling `busibox-frontend` repo cloned as `~/maigent-code/busibox-frontend/`. Nothing in the install path clones it. Docker's auto-create-bind-mount-source then produces a root-owned empty directory that's hard to clean up without sudo.

Fix direction: pull env requirements to `:-` defaults or separate compose files; clone companion repos as part of install; forbid docker from silently creating missing bind-mount sources.

### 4. The install UX is fragmented and internally contradictory

- Issue #006: `busibox-quick` reads the master password via `/dev/tty` with no `--master-password-file` alternative. Can't be automated or driven by an agent.
- Issue #007: Stale vault state from a prior install (vault file + encrypted key) silently mixes with a new setup.
- Issue #014: `INSTALL_UBUNTU_24.04.md` and `install_part2_user.sh` tell users to run `docker compose up -d`; `CLAUDE.md` says never do that.
- Issue #004: `docker restart` doesn't re-read `.env` changes. Doc examples still suggest it.
- Issue #005: Compose project name drifted between `busibox` and `dev-busibox`, leaving orphan volumes & unreusable images.

Fix direction: non-interactive install flags; explicit stale-state detection; one authoritative source of "how to operate services" (probably the CLI) with install docs pointing only there.

## The sequence that actually worked

For posterity, here's the sequence of workarounds applied on 2026-04-23 to get a working install. Numbers refer to issue files.

1. Build the CLI with `./install_part2_user.sh` (clean; part 1 deps already installed from prior run).
2. Delete stale vault state: `~/.busibox/vault-keys/local-development-docker.enc` and `provision/ansible/roles/secrets/vars/vault.local-development-docker.yml` (issue #007).
3. Run `busibox-quick` interactively. It produces `profiles.json` + encrypted vault key + an encrypted vault file full of `CHANGE_ME_*` placeholders (issue #008). Install ends with all 6 bootstrap services failed — `.env.dev` stays empty (issue #009), every compose call hits the vllm `:?` trap (issue #012).
4. Run `scripts/llm/generate-model-config.sh` to produce `model_config.yml` (as the error message suggests — though this alone does not fix the compose error).
5. Decrypt `vault.local-development-docker.yml`, sed-replace every `CHANGE_ME_<KEY>` with a local-dev value, re-encrypt (issue #008).
6. Build `busibox-core`'s `print-vault-password` example binary so we can decrypt the master-encrypted vault key with the known master password, get the 32-char vault password, and export it as `ANSIBLE_VAULT_PASSWORD` for direct ansible runs (issue #006).
7. Manually write `.env.dev` from `.env` + extra vars: `VLLM_MODEL`, `VLLM_PORT`, `VLLM_HOST_PORT`, `CONTAINER_PREFIX`, `COMPOSE_PROJECT_NAME` (issues #009, #012).
8. Clone `https://github.com/jazzmind/busibox-frontend` to `~/maigent-code/busibox-frontend/` (issue #010). First attempt is blocked by root-owned empty dirs docker created; work around with `docker run --rm -v /home/gabe/maigent-code:/w alpine rm -rf /w/busibox-frontend`.
9. Run `ansible-playbook docker.yml` directly, not through the CLI or Makefile, passing `env -u POSTGRES_USER -u POSTGRES_PASSWORD` to strip the shell-inherited vars (issues #001, #011), and explicit extra-vars (`vault_prefix=local-development-docker`, `deployment_environment=local-development-docker`, etc.). Run with `--tags infrastructure,always` → then `apis,always` → then `frontend,always`, each a ~5-20 minute play.
10. Verify: `curl -sk https://localhost/portal/api/health` returns `200 {"status":"healthy","service":"busibox-portal"}`.

Total wall-clock including debugging: several hours. A fresh install with all the above fixes applied in the repo should be under 20 minutes.

## What's NOT fixed here

This directory is diagnosis, not remediation. Individual issues propose fixes but none have been applied to the repo yet. The next step is to convert these into beads / plans, prioritizing:

1. #008 + #009 first — secret substitution + `.env.dev` generation — these two unblock everything else.
2. #001 + #011 together — sanitize the compose subprocess environment.
3. #012 — the 1-character compose change.
4. #010 — clone companion repos during install.
5. Everything else in severity order.
