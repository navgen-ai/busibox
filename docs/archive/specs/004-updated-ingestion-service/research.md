# Research & Technology Decisions

**Feature**: Production-Grade Document Ingestion Service  
**Date**: 2025-11-05

## Executive Summary

This document captures technology decisions and best practices research for implementing a production-grade document ingestion service with multi-vector hybrid search support. Key decisions cover PDF parsing libraries, embedding generation approaches, SSE implementation patterns, and Milvus 2.6+ hybrid search configuration.

## 1. PDF Parsing for Visual Content

### Decision: Marker + TATR (Tables) + Page Images

**Rationale**:
- Marker provides high-quality markdown output preserving document structure
- TATR (Table Transformer) handles complex tables better than generic PDF parsers
- Extracting page images separately enables ColPali visual search
- Dual-track approach (text + visual) provides comprehensive search coverage

**Alternatives Considered**:
- **pdfplumber alone**: Good for simple PDFs but struggles with complex layouts, tables, multi-column text
- **Unstructured.io**: Excellent but heavyweight dependency, similar quality to Marker
- **pypdf2/PyPDF2**: Basic text extraction, misses visual content entirely
- **Adobe PDF Services API**: Expensive, introduces external dependency and latency

**Implementation Notes**:
```python
# Primary path: Marker for text
from marker.convert import convert_single_pdf
markdown_text, images, metadata = convert_single_pdf(pdf_path)

# Table enhancement: TATR for complex tables
from tatr import TableTransformer
tables = TableTransformer.extract_tables(pdf_path)

# Visual path: Extract page images for ColPali
from pdf2image import convert_from_path
page_images = convert_from_path(pdf_path, dpi=150)

# Fallback: pdfplumber for simple PDFs
import pdfplumber
with pdfplumber.open(pdf_path) as pdf:
    text = "\n".join([page.extract_text() for page in pdf.pages])
```

## 2. Multi-Vector Embedding Strategy

### Decision: Dense (liteLLM) + Sparse (Milvus BM25) + Visual (ColPali for PDFs)

**Rationale**:
- **Dense embeddings**: Capture semantic meaning, enable conceptual search
- **Sparse BM25**: Handle exact keyword matches, acronyms, proper nouns
- **ColPali visual**: Capture charts/diagrams/tables without OCR
- Milvus 2.6+ natively supports all three vector types in single collection
- Hybrid search (dense + BM25) outperforms either alone across diverse queries (BEIR benchmark)

**Alternatives Considered**:
- **Dense only**: Misses exact keyword matches, struggles with rare terms
- **BM25 only**: Misses semantic similarity, requires exact term overlap
- **SPLADE learned sparse**: Requires additional model inference, BM25 simpler
- **Separate collections per vector type**: Complicates queries, fusion logic

**Implementation Notes**:
```python
# Dense embeddings via liteLLM
import litellm
dense_embeddings = litellm.embedding(
    model="text-embedding-3-small",
    input=chunks
)

# Sparse BM25 auto-generated by Milvus
from pymilvus import Function, FunctionType
schema.add_function(
    Function(
        name="text_bm25_emb",
        input_field_names=["text"],
        output_field_names=["text_sparse"],
        function_type=FunctionType.BM25
    )
)

# ColPali for PDF pages
from colpali import ColPaliModel
model = ColPaliModel.from_pretrained("vidore/colpali-v1.2")
page_embeddings = model.encode_pages(page_images)  # Returns 128x128 multi-vectors
```

**Model Configuration**:
- **Dense**: `text-embedding-3-small` (1536 dims, fast, cost-effective)
- **Alternative dense**: `bge-m3` (768 dims, multilingual, self-hosted)
- **Visual**: ColPali v1.2 (128 patches × 128 dims = 16,384 dims total per page)

## 3. Server-Sent Events (SSE) Implementation

### Decision: FastAPI SSE + PostgreSQL LISTEN/NOTIFY

**Rationale**:
- SSE is simpler than WebSocket for one-way server-to-client updates
- No need for bidirectional communication (status is read-only)
- PostgreSQL NOTIFY provides efficient pub/sub for status changes
- Avoids polling database every few seconds

**Alternatives Considered**:
- **WebSocket**: Overkill for one-way updates, more complex connection management
- **Polling**: Inefficient, adds database load, higher latency
- **Redis Pub/Sub**: Adds dependency when PostgreSQL already available
- **Long polling**: Less efficient than SSE, more connection overhead

**Implementation Notes**:
```python
from fastapi import Request
from fastapi.responses import StreamingResponse
from sse_starlette.sse import EventSourceResponse
import asyncpg

async def status_stream(file_id: str, request: Request):
    # Establish PostgreSQL connection with LISTEN
    conn = await asyncpg.connect(DATABASE_URL)
    await conn.add_listener('status_updates', 
                            lambda conn, pid, channel, payload: handle_update(payload))
    
    async def generate():
        try:
            # Send current status immediately
            current = await get_current_status(file_id)
            yield {"event": "status", "data": json.dumps(current)}
            
            # Stream updates
            while True:
                # Wait for NOTIFY from worker
                update = await wait_for_update(file_id)
                yield {"event": "status", "data": json.dumps(update)}
                
                if update['stage'] in ['completed', 'failed']:
                    yield {"event": "close", "data": '{"message": "Processing complete"}'}
                    break
        finally:
            await conn.close()
    
    return EventSourceResponse(generate())

# Worker sends NOTIFY when updating status
async def update_status(file_id, stage, progress):
    await conn.execute("""
        UPDATE ingestion_status SET stage = $1, progress = $2, updated_at = NOW()
        WHERE file_id = $3
    """, stage, progress, file_id)
    
    await conn.execute("""
        NOTIFY status_updates, $1
    """, json.dumps({"file_id": file_id, "stage": stage, "progress": progress}))
```

**Dependencies**:
- `sse-starlette` for FastAPI SSE support
- `asyncpg` for async PostgreSQL with LISTEN/NOTIFY
- Keep-alive pings every 30 seconds to prevent connection timeout

## 4. Optimized Text Chunking

### Decision: Semantic boundaries + token limits (400-800 tokens, 10-15% overlap)

**Rationale**:
- 400-800 tokens balances context window vs specificity
- 10-15% overlap ensures boundary chunks don't miss relevant content
- Semantic boundaries (paragraphs, sections) preserve coherence
- Hybrid search benefits from chunks that work well for both BM25 and dense embeddings

**Alternatives Considered**:
- **Fixed 512 tokens**: Ignores semantic structure, may split mid-sentence
- **Paragraph-only**: Variable chunk sizes, some too small (<100 tokens), others too large (>2000)
- **Sentence-based**: Too granular, loses broader context
- **Recursive chunking**: Complex, harder to debug, marginal quality gains

**Implementation Notes**:
```python
import spacy
import tiktoken

nlp = spacy.load("en_core_web_sm")
tokenizer = tiktoken.get_encoding("cl100k_base")

def chunk_with_semantic_boundaries(text: str, 
                                   min_tokens: int = 400,
                                   max_tokens: int = 800,
                                   overlap_pct: float = 0.12):
    doc = nlp(text)
    chunks = []
    current_chunk = []
    current_tokens = 0
    
    for sent in doc.sents:
        sent_tokens = len(tokenizer.encode(sent.text))
        
        if current_tokens + sent_tokens > max_tokens and current_tokens >= min_tokens:
            # Chunk is full, save and start new one with overlap
            chunk_text = " ".join(current_chunk)
            chunks.append(chunk_text)
            
            # Keep last sentences for overlap
            overlap_tokens = int(current_tokens * overlap_pct)
            current_chunk = get_sentences_for_overlap(current_chunk, overlap_tokens)
            current_tokens = count_tokens(current_chunk)
        
        current_chunk.append(sent.text)
        current_tokens += sent_tokens
    
    # Add final chunk
    if current_chunk:
        chunks.append(" ".join(current_chunk))
    
    return chunks
```

**Metadata per chunk**:
- Chunk index (position in document)
- Character offset (for highlighting in original)
- Token count (for debugging, validation)
- Page number (for PDFs, links to ColPali pages)
- Section heading (if detectable, enhances context)

## 5. Milvus 2.6+ Hybrid Search Schema

### Decision: Single collection with multi-field vectors + BM25 function

**Rationale**:
- Milvus 2.6+ supports BM25 functions natively (no external BM25 engine needed)
- Single collection simplifies queries and fusion logic
- Multi-vector field support enables ColPali page embeddings
- All vector types share same metadata (file_id, page_number, etc.)

**Alternatives Considered**:
- **Separate collections**: Requires manual fusion, duplicate metadata
- **OpenSearch for BM25**: Additional service to maintain, network overhead
- **Elasticsearch + Milvus**: Double storage, complex sync

**Implementation Notes**:
```python
from pymilvus import CollectionSchema, FieldSchema, DataType, Function, FunctionType

schema = CollectionSchema([
    # Primary key and metadata
    FieldSchema("id", DataType.VARCHAR, is_primary=True, max_length=64),
    FieldSchema("file_id", DataType.VARCHAR, max_length=64),
    FieldSchema("chunk_index", DataType.INT64),
    FieldSchema("page_number", DataType.INT64),
    FieldSchema("modality", DataType.VARCHAR, max_length=20),  # "text" or "page_image"
    
    # Text content (for BM25)
    FieldSchema("text", DataType.VARCHAR, max_length=65535, enable_analyzer=True),
    
    # Dense embeddings
    FieldSchema("text_dense", DataType.FLOAT_VECTOR, dim=1536),
    
    # Sparse embeddings (BM25) - auto-generated
    FieldSchema("text_sparse", DataType.SPARSE_FLOAT_VECTOR),
    
    # Multi-vector for ColPali
    FieldSchema("page_vectors", DataType.FLOAT_VECTOR, dim=128),  # 128 patches per page
    
    # Metadata
    FieldSchema("metadata", DataType.JSON),
])

# Add BM25 function
schema.add_function(
    Function(
        name="text_bm25_emb",
        input_field_names=["text"],
        output_field_names=["text_sparse"],
        function_type=FunctionType.BM25
    )
)

# Create indexes
collection = Collection("documents", schema)
collection.create_index("text_dense", {"index_type": "HNSW", "metric_type": "COSINE", "params": {"M": 16, "efConstruction": 200}})
collection.create_index("text_sparse", {"index_type": "SPARSE_INVERTED_INDEX"})
collection.create_index("page_vectors", {"index_type": "HNSW", "metric_type": "L2"})
```

**Migration Strategy**:
- Create new collection with hybrid schema
- Migrate existing data (if any) via bulk insert
- Update worker to insert all vector types
- Deprecate old collection after verification

## 6. Redis Streams for Job Queue

### Decision: Redis Streams with consumer groups

**Rationale**:
- Built-in consumer groups prevent duplicate processing
- Automatic message acknowledgment and retry
- Persistent queue (survives Redis restarts if configured)
- Simpler than RabbitMQ/Kafka for this use case

**Alternatives Considered**:
- **Celery**: Heavy dependency, requires broker (Redis/RabbitMQ anyway)
- **RabbitMQ**: Overkill, additional service to maintain
- **SQS/Cloud queue**: Vendor lock-in, network latency
- **Database polling**: Inefficient, higher latency

**Implementation Notes**:
```python
import redis

# Producer (API)
r = redis.Redis(host='ingest-lxc', port=6379)
job_id = r.xadd('jobs:ingestion', {
    'job_id': file_id,
    'file_id': file_id,
    'user_id': user_id,
    'storage_path': storage_path,
    'mime_type': mime_type,
    'created_at': datetime.now().isoformat()
})

# Consumer (Worker)
# Create consumer group (once)
try:
    r.xgroup_create('jobs:ingestion', 'workers', id='0', mkstream=True)
except redis.ResponseError:
    pass  # Group already exists

# Consume messages
messages = r.xreadgroup(
    groupname='workers',
    consumername='worker-1',
    streams={'jobs:ingestion': '>'},
    count=1,
    block=5000  # 5 second timeout
)

# Process and acknowledge
for stream, message_list in messages:
    for message_id, job_data in message_list:
        try:
            process_job(job_data)
            r.xack('jobs:ingestion', 'workers', message_id)
        except Exception:
            # Don't ack - message will be retried
            pass
```

**Configuration**:
- `maxlen` to prevent unbounded queue growth (e.g., 10,000 messages)
- Persistence enabled (`appendonly yes`) to survive restarts
- Multiple workers can consume from same group

## 7. Error Handling & Retry Strategy

### Decision: Exponential backoff for transient errors, permanent failure marking

**Rationale**:
- Transient errors (network timeouts) often resolve quickly
- Exponential backoff prevents thundering herd
- Permanent errors (corrupted files) shouldn't retry indefinitely
- Preserves uploaded files for manual recovery

**Implementation Notes**:
```python
import asyncio
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type

class TransientError(Exception):
    """Network timeout, service unavailable"""
    pass

class PermanentError(Exception):
    """Corrupted file, unsupported format"""
    pass

@retry(
    retry=retry_if_exception_type(TransientError),
    wait=wait_exponential(multiplier=1, min=2, max=60),
    stop=stop_after_attempt(3)
)
async def call_embedding_service(chunks):
    try:
        return await litellm.embedding(chunks)
    except ConnectionError:
        raise TransientError("LiteLLM unavailable")
    except ValueError:
        raise PermanentError("Invalid input chunks")

# Worker processing with error handling
try:
    await process_document(file_id)
except PermanentError as e:
    await mark_failed(file_id, str(e))
    await ack_message(message_id)  # Don't retry
except TransientError as e:
    logger.warning(f"Transient error for {file_id}: {e}")
    # Don't ack - Redis will retry
```

**Error Categories**:
- **Transient**: Network timeout, service unavailable, rate limit
- **Permanent**: Corrupted file, unsupported format, invalid data, missing file

## 9. ColPali for Visual PDF Search

### Decision: vidore/colpali-v1.2 for PDF page embeddings

**Rationale**:
- Late-interaction vision-language model designed for PDF retrieval
- Captures charts, tables, diagrams without OCR
- Multi-vector representation (128 patches per page) enables fine-grained matching
- State-of-art on DocVQA and other PDF retrieval benchmarks

**Implementation Strategy**:
```python
from transformers import AutoProcessor, ColPaliModel
import torch

# Load model (on vLLM or local GPU)
model = ColPaliModel.from_pretrained("vidore/colpali-v1.2")
processor = AutoProcessor.from_pretrained("vidore/colpali-v1.2")

# Process PDF page images
def embed_pdf_pages(page_images):
    inputs = processor(images=page_images, return_tensors="pt")
    with torch.no_grad():
        embeddings = model(**inputs).last_hidden_state
    
    # Returns [num_pages, 128, 128] - 128 patch embeddings per page
    return embeddings.cpu().numpy()
```

**Deployment**:
- Option A: Deploy on vLLM container (requires GPU)
- Option B: Call via liteLLM routing (if vLLM supports ColPali)
- Option C: Separate ColPali service (if volume justifies)

**When to use**:
- PDFs with charts, diagrams, complex tables
- Visual-heavy documents (presentations, reports)
- Skip for text-only documents to save compute

## 11. Chunked Upload for Unlimited File Sizes

### Decision: Multipart upload with stream processing

**Rationale**:
- Enables unlimited file sizes without memory constraints
- Processing can begin before upload completes (faster feedback)
- Memory-efficient as file streams through system
- Standard HTTP multipart upload widely supported

**Alternatives Considered**:
- **Resume uploads**: More complex, requires session management
- **Hard size limit**: Arbitrary restriction, fails for legitimate large files
- **Direct file buffer**: Memory inefficient, fails for large files

**Implementation Notes**:
```python
from fastapi import UploadFile
import hashlib

async def handle_chunked_upload(file: UploadFile):
    """Stream file upload while calculating hash and triggering processing"""
    hasher = hashlib.sha256()
    chunks = []
    
    # Stream to MinIO while hashing
    async with minio_client.put_object_stream(bucket, key) as stream:
        async for chunk in file.file:
            hasher.update(chunk)
            await stream.write(chunk)
            chunks.append(chunk)
            
            # Optional: Begin processing early chunks
            if len(chunks) >= MIN_CHUNKS_FOR_START:
                await trigger_early_processing(chunks[:MIN_CHUNKS_FOR_START])
    
    content_hash = hasher.hexdigest()
    
    # Check for duplicates before full processing
    if existing := await check_duplicate(content_hash):
        return await reuse_vectors(existing, file_id)
```

**Configuration**:
- FastAPI `UploadFile` with streaming enabled
- MinIO multipart upload for files >5MB
- Hash calculation during upload (no extra pass)

## 12. Dynamic Processing Timeouts

### Decision: Size-based timeout tiers (5/10/20 minutes)

**Rationale**:
- Small files shouldn't wait as long as large files
- Large files need more time for complex processing
- Prevents resource exhaustion from runaway jobs
- Easier to debug than single global timeout

**Alternatives Considered**:
- **Fixed timeout**: Too short for large files or too long for small files
- **Linear scaling**: Complex calculation, harder to reason about
- **User-specified**: Abuse potential, requires estimation

**Implementation Notes**:
```python
import asyncio

def get_timeout_seconds(page_count: int) -> int:
    """Calculate timeout based on document size"""
    if page_count < 10:
        return 300  # 5 minutes
    elif page_count <= 50:
        return 600  # 10 minutes
    else:
        return 1200  # 20 minutes

async def process_with_timeout(file_id: str, page_count: int):
    timeout = get_timeout_seconds(page_count)
    
    try:
        async with asyncio.timeout(timeout):
            await process_document(file_id)
    except asyncio.TimeoutError:
        await mark_failed(
            file_id,
            f"Processing exceeded {timeout}s timeout for {page_count}-page document"
        )
        raise
```

**Monitoring**:
- Log timeout events for analysis
- Alert if timeout rate exceeds threshold (e.g., >5%)
- Track processing time distribution per size tier

## 13. Content Deduplication Strategy

### Decision: SHA-256 content hash with storage-level deduplication

**Rationale**:
- SHA-256 collision probability negligible for this use case
- Calculated during upload (no extra pass)
- Each user gets own file record (tracking, permissions)
- Physical storage shared (cost optimization)

**Alternatives Considered**:
- **No deduplication**: Wasted storage, missed performance opportunity
- **File name/size only**: Unreliable, many false negatives
- **Perceptual hashing**: Overkill, not needed for exact duplicates
- **Block-level dedup**: Complex, minimal benefit over file-level

**Implementation Notes**:
```python
import hashlib
from typing import Optional

async def check_duplicate(content_hash: str) -> Optional[dict]:
    """Check if content already processed"""
    existing = await db.fetchrow("""
        SELECT file_id, chunk_count, vector_count
        FROM ingestion_files
        WHERE content_hash = $1
        AND stage = 'completed'
        ORDER BY created_at DESC
        LIMIT 1
    """, content_hash)
    
    return existing

async def reuse_vectors(source_file_id: str, target_file_id: str):
    """Link new file to existing vectors"""
    # Copy file record
    await db.execute("""
        INSERT INTO ingestion_files 
        (file_id, content_hash, user_id, ...)
        SELECT $1, content_hash, $2, ...
        FROM ingestion_files
        WHERE file_id = $3
    """, target_file_id, user_id, source_file_id)
    
    # Link to existing vectors (no copy needed - shared via content_hash)
    await db.execute("""
        UPDATE ingestion_status
        SET stage = 'completed',
            progress = 100,
            message = 'Reused vectors from duplicate content'
        WHERE file_id = $1
    """, target_file_id)
```

**Storage Impact**:
- ~30-50% storage savings for typical workloads
- Near-instant processing for duplicates (<10s vs 2-20min)

## 14. Multi-Language Document Handling

### Decision: Primary language + language array + language-aware chunking

**Rationale**:
- Real-world documents often mix languages
- Primary language enables document-level classification
- Language array enables language-specific filtering
- Language-aware chunking improves search quality

**Alternatives Considered**:
- **Single language only**: Loses information, poor for mixed content
- **Per-chunk language**: More granular but expensive to track
- **"Multilingual" flag**: Not specific enough for search optimization

**Implementation Notes**:
```python
from langdetect import detect_langs
import spacy

def detect_languages(text: str, threshold: float = 0.1):
    """Detect all languages present in document"""
    # Get probabilities for all detected languages
    detections = detect_langs(text)
    
    # Filter by threshold
    languages = [
        {"lang": d.lang, "confidence": d.prob}
        for d in detections
        if d.prob >= threshold
    ]
    
    # Sort by confidence
    languages.sort(key=lambda x: x["confidence"], reverse=True)
    
    return {
        "primary": languages[0]["lang"] if languages else "unknown",
        "all_languages": [l["lang"] for l in languages],
        "confidences": {l["lang"]: l["confidence"] for l in languages}
    }

async def chunk_mixed_language(text: str, languages: list[str]):
    """Chunk with language awareness"""
    chunks = []
    
    for para in text.split("\n\n"):
        # Detect paragraph language
        para_lang = detect(para)
        
        # Load appropriate spaCy model
        nlp = load_spacy_model(para_lang)
        doc = nlp(para)
        
        # Chunk within language boundaries
        for sent_group in group_sentences(doc.sents):
            chunks.append({
                "text": sent_group,
                "language": para_lang,
                "chunk_languages": [para_lang]  # Could have multiple
            })
    
    return chunks
```

**Database Schema**:

### Terminology Clarification

**BM25 Sparse Vectors**: Throughout this project, we refer to "BM25 sparse vectors" or "text_sparse field" interchangeably. These are **automatically generated by Milvus** using a registered BM25 function that analyzes the `text` field. The implementation does NOT require manual BM25 calculation—Milvus handles this internally when data is inserted with the `text` field populated. See INFRA-002 for the function registration code.

**Usage in code**:
- Insert text chunks with `text` field populated
- Milvus automatically generates `text_sparse` field via BM25 function
- Queries can target `text_sparse` for keyword-based retrieval
